# 项目搭建规范、建议与执行计划

## 1. 文档定位

本文目标只有三个：

1. 总结当前项目已经明确的搭建规范。
2. 给出不过度设计的落地建议。
3. 输出一份可以直接执行的分阶段实施计划。

说明：

- 文中的 `demo` 只是占位名称，落地时请替换成真实项目名。
- 本文默认项目形态是“模块化单体”，不是微服务。
- 没有真实业务需求支撑的模块、实体、对象、接口，不提前创建。
- 基础底座已实现完成，首个业务模块待确认名称后进入下一份计划。

## 2. 搭建规范总结

### 2.1 技术基线

- JDK 固定为 17。
- 框架采用 Spring Boot 3.x。
- ORM 采用 MyBatis-Plus。
- 数据库采用 MySQL 8。
- 数据库变更统一由 Flyway 管理。
- 接口文档采用 OpenAPI 3，展示和调试 UI 统一使用 Knife4j。
- JSON 序列化统一使用 Spring Boot 默认 Jackson。

### 2.2 架构基线

- 整体采用清晰分层的模块化单体架构。
- 每个业务模块统一分为 `controller`、`app`、`service`、`infra` 四层。
- 业务调用链作为强约束固定为：`Controller -> AppService -> Domain/Service -> Infra/Mapper`。
- `controller` 负责输入输出和 DTO 适配。
- `app` 层承载 `AppService`，负责编排业务流程，也是事务边界所在层。
- `service` 层承载领域服务或核心业务规则。
- `infra` 层负责数据库访问和外部系统集成，`Mapper` 统一归属 `infra`。
- `controller` 禁止绕过 `AppService` 直接调用 `service`、`infra` 或 `mapper`。
- `AppService` 禁止直接面向 Web DTO 编写底层持久化逻辑，持久化访问统一下沉到 `service` / `infra` 链路。
- 分包必须按业务领域划分，不能按技术类型散乱拆包。

### 2.3 数据规范

- 默认使用逻辑删除，字段名统一为 `deleted`。
- 默认所有业务数据表都必须包含 `tenant_id`。
- 平台级全局表如果不参与租户隔离，必须在设计和 DDL 中显式标注为“非租户表”。
- 所有非租户表都必须加入 `TenantLineHandler` / `TenantLineInnerInterceptor` 的忽略清单，禁止只靠人工约定绕过租户插件。
- 所有业务表默认包含 `create_time`、`update_time`、`create_by`、`update_by`、`deleted`。
- `create_time` 和 `update_time` 由数据库自动维护。
- `create_by` 和 `update_by` 通过 MyBatis-Plus 自动填充。
- 多租户隔离通过 MyBatis-Plus 的 `TenantLineInnerInterceptor` 统一实现。

### 2.4 API 与编码规范

- 统一响应结构为 `{"code":200,"msg":"ok","data":...}`，响应对象命名为 `R`。
- 所有接口统一使用 `POST`。
- URL 统一使用 `/api/{模块名}/{资源名}/{动作}`。
- Web 层请求和响应对象必须使用独立 DTO，不能复用数据库实体。
- 请求对象统一命名为 `XxxReqDTO`，响应对象统一命名为 `XxxRspDTO`。
- 分页统一使用 `PageReqDTO` 和 `PageResult<T>`。
- 全局异常统一转换为标准 `R<T>` 响应。
- 日期时间统一格式为 `yyyy-MM-dd HH:mm:ss` 和 `yyyy-MM-dd`。

### 2.5 模块解耦规范

- 主数据模块和账号权限模块是基础模块，其他业务模块按需接入。
- 模块之间禁止直接依赖对方实现包。
- 需要同步返回值的跨模块调用，通过独立 `-api` 契约包完成。
- 不需要返回值的事后通知，统一通过 Spring `ApplicationEvent` 解耦。
- 租户、当前登录人、全局常量、通用事件定义统一下沉到 `demo-core`。

### 2.6 扩展能力规范

- 文件存储需支持本地文件系统和 MinIO 两种实现，通过配置切换。
- 关联 ID 到名称的展示翻译，不在 SQL 中写大量 `LEFT JOIN`，统一走翻译机制。
- 简单 CRUD 查询和复杂查询要分开处理。
- 影响后端逻辑分支的状态值用 Enum，仅用于展示和筛选的可变内容用 Dict。
- 导出场景优先使用独立 `ExportDTO`，不要污染接口响应对象。

## 3. 克制性建议

下面这些建议的目标只有一个：保留规范约束，但避免一开始把系统做重。

### 3.1 先做模块化单体，不拆微服务

- 当前规范强调的是模块边界和可拔插部署，不等于必须先拆成多进程微服务。
- 更合适的起步方式是一个 Maven 多模块工程，加一个统一启动模块。
- 这样既能保留模块隔离，又能降低开发、调试、部署复杂度。

### 3.2 初始模块控制在最小集合

建议第一阶段只创建以下模块：

- `demo-boot`：启动模块，负责装配和运行。
- `demo-core`：全局共享能力。
- `demo-system`：账号、权限、租户、审计等基础能力。
- `demo-mdm`：主数据基础能力。
- 第一个真实业务模块：只有在业务需求明确后再创建。

不建议一开始就创建：

- 多个业务空模块。
- 每个模块配套的 `-api` 包。
- `provider`、`openapi`、`adapter`、`facade` 等大量空包。

### 3.3 对象模型保持最小化

第一阶段只保留三类核心对象：

- `Entity`：数据库持久化对象。
- `ReqDTO` / `RspDTO`：Web 层请求响应对象。
- `XxxQuery`：只在复杂查询场景下引入。

不建议一开始同时引入：

- `VO`、`BO`、`DO`、`Param`、`Form`、`Command`、`QueryDTO` 等多套近义对象。

### 3.4 能力先做“够用版本”

- 测试基础设施、参数校验、日志输出、多环境 profile、密码哈希属于第一阶段必备基建，不后移。
- 翻译引擎先支持当前真实需要的类型，例如 `USER`、`DICT`。
- 动态查询先支持等值、模糊、区间、`in` 这类高频能力。
- 文件存储在没有真实上传需求前不抽象公共 SPI，出现首个上传场景后先落本地实现，再按需要补 MinIO。
- `OperationLog` 不在零业务阶段做“注解 + 切面 + 事件 + 异步落库”全闭环，只保留后续接入位和最小日志能力。
- 依赖版本不要使用 `LATEST`、`RELEASE` 或动态范围，统一锁定明确版本号。

### 3.5 避免“抽象先行”

- 不要为了“以后可能会复用”先造通用业务基类。
- 不要为了“将来可能跨模块调用”提前给所有模块建 `-api`。
- 不要为了“将来可能导出”提前在所有 `RspDTO` 中加入翻译冗余字段。
- 不要在没有真实场景前引入 MQ、分布式事务、外部工作流引擎。

## 4. 建议的最小工程结构

```text
java-demo/
├── pom.xml
├── README.md
├── docs/
│   └── create.md
├── demo-boot/
├── demo-core/
├── demo-system/
├── demo-mdm/
└── demo-{biz}/                # 只有出现真实业务需求时再创建
```

### 4.1 根工程

- 根 `pom.xml` 负责依赖版本统一、插件版本统一、模块聚合。
- 不在根工程堆业务代码。

### 4.2 启动模块 `demo-boot`

- 只负责 Spring Boot 启动、配置装配、扫描模块 Bean。
- 放置 `application.yml`、`application-dev.yml`、`application-test.yml`、Flyway 扫描配置、Knife4j 配置、Actuator 配置。
- 不写业务逻辑。

### 4.3 核心模块 `demo-core`

`demo-core` 必须保持“薄核心”，只承载真正全局通用、跨模块稳定复用的基础设施，不承载业务语义。

建议从第一天就拆成清晰的内部包，避免后续所有公共代码都堆进一个目录：

```text
com.demo.core
├── web                # R<T>、统一分页对象、通用 Web 返回约定
├── exception          # 错误码、异常定义、全局异常处理
├── validation         # Bean Validation 扩展与统一校验异常转换
├── jackson            # Jackson 全局配置
├── trace              # TraceId、MDC、请求链路追踪
├── tenant             # TenantContext、非租户表清单、租户插件配置
├── security           # UserContext、密码编码器等安全基础设施
├── mybatis            # MyBatis-Plus、分页、逻辑删除、自动填充配置
├── logging            # logback 约定、日志切面基础能力
├── config             # 跨模块基础配置
└── enums              # 全局公共枚举
```

建议先放入以下通用能力：

- `R<T>` 响应包装。
- 错误码定义。
- 全局异常处理。
- Bean Validation 集成与校验异常统一处理。
- Jackson 全局配置。
- `PageReqDTO`、`PageResult<T>`。
- 租户上下文、用户上下文、`TraceId` 上下文。
- 密码编码器等安全基础设施。
- MyBatis-Plus 公共配置。
- 自动填充处理器。
- 逻辑删除、多租户、分页拦截器配置，以及非租户表忽略清单维护点。
- logback 日志配置约定，确保 `traceId` 可输出。
- 全局公共枚举。

错误码约定从第一天就固定：

- 成功响应使用 `code = 200`。
- 默认失败响应使用 `CommonErrorCode.FAILED(500, "操作失败")`。
- 公共错误码放在 `demo-core.exception`，使用稳定通用码，例如 `400 / 401 / 403 / 404 / 429 / 500`。
- 业务模块错误码放在模块内 `enums`，例如认证 `2001xxx`、主数据 `3001xxx`。
- 业务异常优先通过 `ErrorCode` + `BizException` 抛出，不在 service / app 中散落裸数字。
- 未捕获系统异常统一映射到公共内部错误码，不向前端暴露堆栈语义。

不要提前放入以下内容：

- 没有真实消费者的文件存储 SPI。
- 没有真实业务场景的翻译引擎实现。
- 零业务阶段就完整落库的 `OperationLog` 全链路。

### 4.4 业务模块内部目录

每个业务模块默认只保留以下目录：

```text
com.demo.{module}
├── controller
├── app                 # AppService，事务边界与流程编排
├── service             # Domain/Service，领域规则
├── infra
│   └── mapper          # Mapper 统一归 infra
└── enums                      # 只有模块内确实存在私有枚举时再创建
```

以下目录只在需要时创建：

- `service/query`：复杂查询时创建。
- `provider`：对兄弟模块提供内部服务时创建。
- `openapi`：对外第三方开放时创建。
- `convert`：对象转换逻辑足够多时再独立。

### 4.5 API 契约模块

`demo-xxx-api` 只在出现“模块 A 必须同步调用模块 B 且需要返回值”的真实场景后才创建。

它的职责应当非常单一：

- 定义 `XxxProvider` 或 `XxxApi` 接口。
- 定义内部调用使用的 DTO。
- 不放实现。

## 5. 详细可执行计划

### 5.1 阶段一：建立工程骨架

目标：先把工程变成一个可以启动、可以管理依赖、可以继续开发的标准 Maven 多模块项目。

执行项：

1. 新建根 `pom.xml`，定义父工程、模块聚合、JDK 17、依赖版本和插件版本。
2. 新建 `demo-boot`、`demo-core`、`demo-system`、`demo-mdm` 四个模块。
3. 在 `demo-boot` 中创建 Spring Boot 启动类和基础配置文件。
4. 统一确定包名、模块名、数据库命名规范、配置文件命名规范。
5. 建立 `dev`、`test` 两套最小 profile 配置骨架。
6. 加入 Spring Boot Actuator 基础依赖和健康检查端点。

完成标准：

- 根工程可以成功解析依赖。
- `demo-boot` 可以空启动。
- `dev` / `test` profile 可以正常切换。
- 健康检查端点可访问。
- 模块引用关系清晰，没有循环依赖。

### 5.2 阶段二：搭建 `demo-core`

目标：把所有“每个模块都会依赖”的共性能力先打牢。

执行项：

1. 实现统一返回对象 `R<T>`。
2. 建立错误码约定和全局异常处理器。
3. 集成 Bean Validation，并统一处理参数校验异常。
4. 实现 `PageReqDTO`、`PageResult<T>`。
5. 实现 Jackson 日期时间全局格式化。
6. 实现 `TraceId` 生成和 MDC 注入。
7. 提供 `logback-spring.xml` 或等价日志配置，确保日志中可输出 `traceId`。
8. 实现租户上下文和当前用户上下文。
9. 配置 MyBatis-Plus 的分页、逻辑删除、多租户拦截器。
10. 显式维护非租户表忽略清单，避免平台级全局表误加租户条件。
11. 实现 `MetaObjectHandler`，自动填充 `create_by`、`update_by`。
12. 建立测试基础设施：`JUnit 5`、`Spring Boot Test`，以及至少一个从 Controller 到数据库的集成测试样板。
13. 加入开发态最小 CORS 配置。
14. 放入全局公共枚举，例如 `YesNoEnum` 这类基础枚举。

完成标准：

- 所有模块都能直接复用 `demo-core` 公共能力。
- 新增接口时无需重复写统一响应、分页、异常处理样板代码。
- 多租户、逻辑删除、审计填充的基础配置已经可用。
- 参数校验失败、业务异常、系统异常都能稳定返回统一响应。
- 测试可以在 `test` 环境下稳定运行。
- 日志中可以稳定输出 `traceId`。

### 5.3 阶段三：搭建数据库与迁移机制

目标：先把数据库演进流程规范化，再进入业务开发。

执行项：

1. 在工程中集成 Flyway。
2. 确定迁移脚本目录规则，例如按模块分类维护。
3. 建立基础表设计约束模板，确保所有业务表默认包含公共字段。
4. 建立租户字段约束模板，确保默认所有业务数据表都有 `tenant_id`。
5. 为平台级全局表建立“非租户表”显式标识规范，并同步维护到 `TenantLineHandler` 忽略清单。
6. 显式配置 HikariCP 的关键参数，例如连接池大小、超时和必要的泄漏检测阈值。
7. 为基础模块编写首批迁移脚本。

建议首批只落以下最小表集合：

- 租户表。
- 用户表。
- 角色表。
- 角色授权关系表。
- 用户角色关系表。
- 操作日志表。
- 主数据里真正马上会用到的字典或组织表。

完成标准：

- 本地数据库可以通过 Flyway 一键初始化到最新版本。
- 表结构满足公共字段、逻辑删除、多租户约束。
- 非租户表已经被显式识别并加入租户插件忽略清单。
- 后续新增表不再允许手工改库。

### 5.4 阶段四：搭建 `demo-system`

目标：先打通系统底座，而不是先做业务模块。

执行项：

1. 实现租户、用户、角色、权限的最小闭环。
2. 实现登录接口和最小鉴权能力。
3. 实现当前登录人上下文注入。
4. 明确密码哈希方案，默认使用 `BCrypt` 或 `Argon2`，禁止明文或可逆存储。
5. 为用户、角色、权限建立最小管理接口。
6. 在开发态提供最小必要的跨域配置。

建议控制范围：

- 第一个版本先把“能登录、能鉴权、能区分租户”做好。
- 不要一开始就做复杂 RBAC 模型扩展、多维权限、数据权限编排。

完成标准：

- 用户可以完成登录。
- 请求链路上可以拿到租户 ID 和当前用户信息。
- 密码在库中以安全哈希方式存储。

### 5.5 阶段五：搭建 `demo-mdm`

目标：把其他业务模块都会依赖的主数据底座建立起来。

执行项：

1. 只实现首批真实需要的主数据能力。
2. 如果当前业务明确需要字典，就实现字典类型和字典项。
3. 如果当前业务明确需要组织架构，就实现组织或部门主数据。
4. 如果当前业务暂时不需要供应商、仓库、客户等主数据，就不要预建。

建议控制范围：

- `demo-mdm` 不是“所有主数据都先建完”，而是“先建公共底座里马上要用的主数据”。
- 翻译引擎只有在真实页面出现 ID 转名称展示需求后再落第一版。

完成标准：

- 首个业务模块依赖的基础主数据已经可用。
- 主数据接口和业务接口之间边界清晰。

### 5.6 阶段六：接入第一个真实业务模块

目标：用一个真实业务模块验证整套规范是否能落地，而不是停留在框架设计层面。

执行项：

1. 只在业务需求明确后创建第一个业务模块。
2. 该模块内部严格按 `Controller -> AppService -> Domain/Service -> Infra/Mapper` 链路落地。
3. 简单单表 CRUD 先使用最小查询能力。
4. 只有出现复杂联表、报表或内部契约场景时，再引入 `XxxQuery`。
5. 只有出现跨模块同步调用需求时，再拆出该模块的 `-api` 包。
6. 只有出现事后通知需求时，再补充领域事件。
7. 在首个真实业务模块中再补 `@OperationLog` 的业务接入和落库闭环。

完成标准：

- 第一个业务模块能完成完整 CRUD 或核心业务闭环。
- 没有为了“以后可能会用”而引入额外空包和空层。

### 5.7 阶段七：补齐文件、翻译、导出等扩展能力

目标：只在基础业务跑通之后，再补扩展能力。

执行项：

1. 当出现真实上传需求时，实现本地文件存储。
2. 在进入联调或部署阶段前补齐 MinIO 实现。
3. 当页面确实存在 ID 转名称展示需求时，实现翻译能力。
4. 当导出场景出现时，再创建独立 `ExportDTO` 和导出处理器。
5. 当跨模块同步调用出现时，再补对应 `-api` 契约模块。

完成标准：

- 每项扩展能力都由明确业务场景驱动。
- 没有提前引入暂时不用的表、实体、接口和流程。

### 5.8 阶段八：验证与收口

目标：把“能开发”提升到“可持续开发”。

执行项：

1. 验证空工程启动是否正常。
2. 验证 Flyway 迁移是否正常。
3. 验证统一响应和全局异常是否正常。
4. 验证多租户隔离是否正常。
5. 验证逻辑删除和自动填充是否正常。
6. 验证 `TraceId` 和日志输出是否正常。
7. 验证首个业务模块接口闭环是否正常。
8. 验证测试样板可以稳定运行。
9. 验证非租户表不会被租户插件误拦截。

完成标准：

- 工程可以稳定启动。
- 数据库可以稳定迁移。
- 基础规范已经被真实接口验证，不只是停留在文档层面。

## 6. 明确不提前做的事项

为了避免过度设计，以下内容在没有真实需求前不建议进入首版：

- 微服务拆分。
- MQ。
- 分布式事务。
- 全量业务模块预建。
- 全量 `-api` 契约包预建。
- 大而全的动态查询 DSL。
- 大而全的翻译类型体系。
- 提前定义大量导出 DTO。
- 为复用而复用的抽象基类体系。
- 没有业务支撑的主数据实体。

## 7. 最终落地原则

这套规范的核心不是“把架构名词堆满”，而是：

- 边界要清楚。
- 公共能力要统一。
- 首版范围要克制。
- 每新增一个模块、实体、对象、接口，都必须有当前业务理由。

如果按这个原则推进，项目既能满足后续扩展和模块解耦要求，也不会在第一阶段因为过度设计而失速。
